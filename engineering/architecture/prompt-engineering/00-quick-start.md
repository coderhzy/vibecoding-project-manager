# 00 - 快速入门：Agent 提示词设计规范

> 核心理念：用最少的 token 达到目标，结构清晰比解释详尽更重要。

## 设计原则

### 1. 简洁优于冗长

2025 年最佳实践：结构化的简洁提示词优于冗长的详细指令。

| 对比 | Token 数 | 效果 |
|------|----------|------|
| 详细指令 | 2,500 | 日成本 $3,000 |
| 结构化提示 | 212 | 日成本 $706（-76%） |

### 2. 信任模型理解能力

- 不需要过度解释"怎么做"
- 只需要说清楚"做什么"和"输出什么"
- 对于 reasoning 模型，给高层目标，让模型自己推导

### 3. 低创造性任务用低温度

提取、整理、归纳类任务：
- `temperature=0` 或 `0.1`
- 关闭 reasoning
- 选用快速模型

## 角色设计框架

### 1. 工作场景（Work Context）

```markdown
# 工作场景
你工作在一个短视频内容生产流水线中。
流程：用户需求 → 选题师设计选题 → 用户确认 → 文案师撰写文案
你的位置：选题确认后、文案撰写前的交接环节。
```

### 2. 任务（Task Context）

```markdown
# 任务
阅读选题阶段的完整对话，整理出创作清单给文案师。
文案师只会看到你的输出，看不到之前的对话。
```

### 3. 领域知识（Domain Context）

```markdown
# 领域知识
- 选题师会用【素材#X】格式标注引用的素材
- 用户确认选题时会说"就这个"、"B吧"、"确认"等
- 用户可能提出创作要求："口语化"、"不要硬广"等
```

### 4. 输出风格（Style + Tone + Intent）

```markdown
# 输出风格
- Markdown 分块，标题清晰
- 中立客观，不加修饰
- 只提取对话中明确出现的内容，不推测、不补充
```

## 提示词模板

```markdown
# 工作场景
{描述业务流程和当前位置}

# 任务
{具体做什么，输出给谁}

# 领域知识
{业务规则、识别模式、具体示例}

# 输出风格
{格式、语调、边界约束}

# 输入
{input}

# 输出
{期望的输出结构}
```

## 常见错误

### 过度解释

```
❌ 错误：
请逐项检查并提取：
□ 确认选题：用户最终选了哪个？
□ 素材引用：选题师标注了哪些素材？
...

✅ 正确：
# 领域知识
- 选题师会用【素材#X】格式标注引用的素材
- 用户确认选题时会说"就这个"、"B吧"、"确认"等
```

### 缺少工作场景

```
❌ 错误：
你是一个交接单整理员。

✅ 正确：
# 工作场景
你工作在一个短视频内容生产流水线中。
流程：用户需求 → 选题师 → 用户确认 → 文案师
你的位置：选题确认后、文案撰写前。
```

### 使用"想象"表述

```
❌ 错误：
想象你是一个专业的内容编辑...

✅ 正确：
你是选题→文案阶段的交接员。
```
